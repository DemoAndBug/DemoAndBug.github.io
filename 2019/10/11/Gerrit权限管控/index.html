Gerrit权限管控

Gerrit项目权限介绍

  1. 在gerrit中添加了以下用户：

gerrit 用户： 服务管理员用户
manager 用户： 项目管理员
jenkin 用户： Verifier用户 对代码进行校验 通常由jenkins自动化使用 也可以配置相应人员使用
reviewer 用户：项目代码reviewer 人员，可以 +2 -2
simon 用户： 项目开发人员

  1. 在gerrit中创建用户组

DevopsDevelops ： devops项目开发人员，用户组成员simon
DevopsManager : devops项目管理员人员，用户组成员manager
DevopsReviewer： devops项目代码reviewer人员，用户组成员reviewer
DevopsVerifier： devops项目代码verifier人员，用户组人员jenkins

Non-Interactive Users： 非交互人员组，主要用户jenkins自动化
Manager: 总管理员组，不局限与某一个project
Testers：测试人员组

所有用户组创建默认包含gerrit管理员。

  1. 在gerrit中创建项目

创建项目devops 项目 与 frontend项目 前端项目模板；
我们会看到新建项目会默认继承All-Projects的权限，修改权限的时候慎重，不要直接修改All-Projects组的权限，因为这个是所有项目的依赖权限组，修改了以后，所有的项目权限都会跟着发生变化。

每个project可以添加自己的Reference,每个Reference可以配置自己的permission；

  1. 项目权限

1、Reference: refs/*
主要配置读取权限 Read

2、Reference: refs/for/* 拥有添加Patch的权限。
主要配置patch权限 Add Patch Set

3、Reference: refs/for/refs/* 拥有Push和Push Merge Commit权限。
主要配置Push 和 Push Merge Commit 权限

4、Reference: refs/heads/* 重点配置项 它是git存储库的所有分支
主要配置Push、Review、Verified、Submit 、Edit Topic Name permission等

5、Reference:refs/tags/* 可以执行所有的标记功能。
Reference:refs/meta/config 管理元数据的，系统配置相关权限，也可以让别人自定义规则。

Gerrit项目管控

  1. 修改All-Projects refs/* read 选项权限,禁止注册用户和匿名用户，这样普通用户就看不到所有project了
  2. 对特定project 允许特定groups查看和提交代码
  3. 安装gitweb后，需要普通用户查看gitweb页面的，需要开启refs/meta/config read权限即可，默认只有管理员有权限查看

项目创建模式可以参考Project图：

权限功能类似与子类继承父类，公共配置可以抽取到模板中进行配置，差异性在各自的ACL（Access Control List）单独配置。

用户组创建模式参考Group图：

Administrators 与 projectManager之间创建一个总管理员组（manager），权限介于Administrators与ProjectManager之间。然后Group主要配置如下

1、每个项目的projectManager组 需要 Included manager组；

2、每个项目的projectReviewer组 需要 Included projectManager组；

3、每个项目的projectVerifier组 也需要 Included projectManager组；

4、每个项目的projectDevelops组 需要 Included 以上所有组；

5、project权限配置，截图演示是继承All-Projects，实际项目创建需先建立模板然后继承配置；切记不要大肆修改All-Projects权限，这个是会影响所有项目。
